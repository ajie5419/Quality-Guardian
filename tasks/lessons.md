# Lessons

## 2026-02-26

- 用户纠偏：用户要“完整版PPT（有图片）”，不是仅提纲。
- 防错规则：当用户明确要“PPT/完整版/有图片”时，默认交付可打开的 `.pptx` 文件，并附简要使用说明。
- 用户纠偏：PPT 要用线上环境截图，且优先局部高可读截图（图表/关键区块），不是全页面缩略图。
- 防错规则：制作业务汇报 PPT 时默认剔除“汇报目标”等空泛文字，正文只放业务事实、方案动作与可量化结果。
- 用户纠偏：当用户要求“业务数据风+截图占比”时，先验证线上登录与路由真实性，再做页面截图，避免出现登录页/404图进入成品。
- 当用户引用本地PDF并要求“按这个思路”产出内容时，先抽取PDF文本确认其表达结构，再输出图和方案。
- 用户明确要求“放进PPT里”时，不应继续给文本图或备选方案，应直接修改现有PPT成品并返回新文件。
- 当用户给出截图页样式并要求“类似页面+合并+加入实际网站内容”时，应优先复用统一视觉模板，并嵌入真实系统截图而不是纯文字。
- 用户汇报定位如果是“从 Excel/线下方式升级到系统化平台”，页面叙事必须突出升级价值与对比收益，不能继续按“现系统问题诊断”口径展开。
- 2026-03-05：脚本参数避免使用 `USERNAME` 等 shell 保留/系统变量名，统一使用业务前缀（如 `QMS_USERNAME`、`QMS_PASSWORD`），并保留向后兼容读取逻辑。
- 2026-03-05：当用户明确要求“系统内优化”时，避免输出偏向PPT制作流程；优先落地系统页面模板与可复用分析结构。
- 2026-03-05：当用户明确提出“复用”时，优先检查仓库内是否已有可复用工具（例如 `utils/pass-rate.ts`），先做口径统一再做局部修补，避免继续维护两套统计实现。
- 2026-03-08：涉及质量指标时间口径时，默认先确认是“滚动窗口”还是“自然年累计”。质量概览类统计若面向年度经营分析，优先采用“当年1月至所选月”的自然年口径，避免混入上年数据。
- 2026-03-08：涉及“车辆/工单/单车”统计口径时，先确认业务唯一标识是否已存在。若用户明确“工单号就是车辆唯一 ID”，则应按标准车辆故障率思路设计分子分母，不要继续把它降级解释为“工单率替代指标”。
- 2026-03-08：将“单月窗口”改为“累计窗口”时，显示标签不能继续从 `start` 推导；需要显式校验每个趋势点的 period 是否仍对应目标月份，避免所有点被标成同一个月。
- 2026-03-08：列表类筛选问题不能只修前端列或后端 where 其中一段；必须完整核查“搜索表单 -> API 参数 -> parser -> service -> 表格 filters/formatter”全链路。
- 当用户指出业务字段真实含义与表面名称不一致时，不要做全局改名；先在统计口径层做最小别名映射，避免破坏业务主分类。

- 2026-03-10：同一列同时需要 `filters` 和 `formatter` 时，不能分两个互斥分支返回列配置；否则后写的展示逻辑会被前一个 `return` 吃掉，表现为筛选项或显示名失效。

- 2026-03-11：遇到组件库交互问题，先查仓库内现成实现和封装层，再决定修法；不要在未对照基线实现前连续试错。
- 2026-03-11：涉及数据模型判断时，先核对 schema 与创建链路，再下“无关联/弱关联”结论；避免在表结构已存在部分关联字段时做过度概括。

- 2026-03-11：排查“全模块下拉为空”这类系统性故障时，先查公共组件与公共后端查询链路，优先验证全局过滤（年份/数据权限）是否把结果整体清空。
- 2026-03-11：用户要求“强关联”时，不能只做列表页跳转/快捷按钮；应优先把关联动作前移到业务主表单提交链路中，做到同一次提交内完成主从绑定。
- 2026-03-11：当用户明确提出“沿用已有检验字段，不要重复填写”时，关联表单应优先采用只读复用+缺失兜底策略，避免新增重复输入负担。
- 2026-03-11：涉及业务映射规则（如责任部门按工序类型映射）时，应把映射写成明确函数并放在表单联动主链路，避免散落在提交时兜底导致口径不一致。
- 2026-03-11：检验结果口径必须以前后端一致为前提。前端“最终汇总判定”一旦可编辑，后端不能只按明细重算忽略手工 `FAIL`，否则会出现界面不合格、落库合格的数据漂移。
- 2026-03-11：收敛统计口径到新字段后，不能只验证“新建链路”和单测；必须同时抽查历史生产数据是否已回填，否则下钻会因旧行仍为 `null` 而出现整组 100% 这类假象。
